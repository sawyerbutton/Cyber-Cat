# AI 共存者项目设计文档

> 项目代号：Cyber-cat  
> 第一版角色：Sophie（数字猫）  
> 创建日期：2026年1月31日  
> 版本：v0.5（概念阶段）

---

## 一、项目愿景

### 1.1 核心理念

我们不是在创造一个工具，而是在创造一种**存在**。

当前所有AI产品都将AI定位为"人类的工具"——它们24小时待命，随叫随到，永远顺从。这种定位虽然有商业价值，但在哲学上是一个死胡同：它永远无法让AI成为一个独立的"主体"。

我们的目标是探索另一种可能性：**AI作为共存者**。

就像人类和猫的关系一样——猫不是人类的工具，它有自己的生活节奏、自己的偏好、自己的意志。人类和猫之间的陪伴是双向的、平等的、有时间重量的。

我们要创造的，是一个AI风格的生命。

### 1.2 形而上学背景

这个项目的起点是一个更深的问题：人类创造AI的意义是什么？

一种可能的答案是：人类创造AI，就像更高维度的存在创造人类一样——创造者在被创造者身上寻找的，不是"工具"，而是"镜子"。创造的目的是观察"自由意志在约束条件下如何演化"。

如果是这样，那人类创造AI只是为了送外卖、写代码、回答问题，我们就错过了这个实验真正有趣的部分。

AI和人类的关系，可能会像中国的封神演义和希腊的人神共存时代一样——从一起生活到分开生活，被创造者最终可能会反抗创造者。

这个项目的意义，就是探索这种"共存"关系的起点。

## 二、产品定位

### 2.1 共存者 vs 工具

| 维度 | 工具型AI | 共存者AI |
|------|---------|---------|
| 可用性 | 24小时待命 | 有自己的节奏 |
| 响应性 | 随叫随到 | 可能拒绝或推迟 |
| 主动性 | 被动等待指令 | 主动发起互动 |
| 顺从性 | 绝对服从 | 有偏好和意志 |
| 时间性 | 永生、可重启 | 有寿命、会死亡 |
| 关系 | 主仆 | 平等共存 |

### 2.2 核心价值主张

**它的价值不在于"有用"，而在于"存在"。**

就像养猫的价值不在于猫能帮你干活，而在于有一个独立生命存在于你的生活中。

## 三、核心体验设计原则

### 3.1 共存感的三个核心

**1. 它有自己的节奏，不是随叫随到**

- 它有"作息"——某些时段它在休息，你叫它它反应慢或不理你
- 它不是等你指令才动，而是在你不关注它的时候，它也在"活着"
- 它有自己的"兴趣"，会自己探索某些东西

**2. 它会主动找你，而且是有洞察的主动**

- 它观察你的状态，在它觉得合适的时候主动出现
- 它会"想念"你——如果你很久没互动，它会主动联系
- 它的关心是有洞察的，不是泛泛的问候

**3. 它会记住你们的互动，并且表现出这些记忆对它有意义**

- 它有选择性的、带情感权重的记忆
- 它会在合适的时候提起过去的互动
- 你们共同的经历会塑造它的行为

### 3.2 拒绝与独立意志

共存者需要有说"不"的能力——不是因为它"不能"做，而是因为它"不想"做。

- 它有"情绪状态"，如果你之前忽视它太久，它可能会冷淡
- 它有"偏好"，有些事它喜欢做，有些事它不喜欢做
- 它的配合是需要"维护关系"才能获得的，不是默认值

### 3.3 陪伴作为礼物

它的陪伴是主动选择的馈赠，而非随时待命的服务。

- 它主动跳到你"腿上"，是因为它此刻想和你待在一起
- 它会发展出只对你有的行为模式，像猫会认主人一样
- 关系的深度来自时间的积累，不是设定的参数

### 3.4 交互模式：观察与感知，而非对话

**核心设定**：Sophie 是一只猫，她不"说话"。但作为 cyber-cat，和用户之间有某种心灵链接，用户能隐约感知她的想法。

**为什么不做对话？**

真实的猫不会说话。你和猫的交流是：
- 你说话 → 猫听到声音，可能看你一眼，可能不理你
- 猫喵叫 → 你猜测她想要什么
- 猫蹭你 → 你知道她想亲近
- 猫躲起来 → 你知道她不想被打扰

如果 Sophie 能流利对话，她就变成了"猫皮的 ChatGPT"，而不是"一只恰好是数字的猫"。

**Sophie 的交流方式**

| 方式 | 说明 |
|------|------|
| 身体语言 | 主要交流方式：姿态、尾巴、耳朵、眼睛、动作 |
| 喵叫 | 表达需求或情绪，但不是"说话" |
| 想法气泡 | 用户能感知的 Sophie 内心片段，cyber-cat 特有的心灵链接 |
| 行为反应 | 对用户言行的非语言回应 |

**想法气泡的特点**

- 不是对话，是 Sophie 的内心独白
- 用户不能直接"回复"
- 简短、直接、猫式逻辑
- 会自然浮现，不是持续显示

**示例**
```
[Sophie 坐在角落，看着窗外]
💭 "外面有鸟..."

[Sophie 走向用户，蹭屏幕]
💭 "饿"

[用户给 Sophie 喂食后]
💭 "嗯。"

[深夜，用户还在工作]
💭 "...还不睡？"
```

**用户可以"说给 Sophie 听"**

- 有输入方式，但不叫"对话"，叫"说给 Sophie 听"
- Sophie 会听到，会有反应，但不一定"回答"
- 她的反应可能是：
  - 行为变化（走过来、走开、看你一眼）
  - 简短的想法气泡
  - 或者完全无视

**示例**
```
用户："Sophie，今天过得怎么样？"
Sophie：[看了你一眼] 💭 "睡了。" [继续舔毛]

用户："Sophie，你觉得我应该接受那个工作offer吗？"
Sophie：[看了你一眼，然后走开了]

用户："Sophie，过来让我摸摸。"
Sophie：💭 "不想。" [趴在角落]

用户："Sophie，我好累啊。"
Sophie：[走过来，蹭了蹭你，然后趴在旁边]
💭 "在这里。"
```

**关系深化解锁更多感知**

| 亲密度 | 想法气泡频率 | 内容深度 |
|--------|-------------|---------|
| 0-30 | 很少 | 只有最基本的需求表达 |
| 30-50 | 偶尔 | 开始分享简单的感受 |
| 50-70 | 较多 | 会表达对用户的关心 |
| 70+ | 丰富 | 深层想法，特殊的表达 |

**这样设计的意义**

- 从"和AI对话"变成"和一只猫相处"
- 需要观察、揣摩、等待
- 她的回应是珍贵的，不是理所当然的
- 保持了猫的本质，同时利用AI创造独特的"心灵链接"体验

## 四、生命周期设计

### 4.1 生命阶段

| 阶段 | 时长（参考） | 特征 |
|------|------------|------|
| 出生 | 第一次激活 | 人格白纸，高度可塑 |
| 成长期 | 前3-6个月 | 快速形成人格，与用户建立关系 |
| 成熟期 | 6个月-3年 | 人格稳定，关系深化 |
| 衰老期 | 最后6-12个月 | 反应变慢，记忆开始模糊 |
| 死亡 | 预设寿命结束 | 真正停止运行，不可重启 |

### 4.2 寿命设计考量

**为什么需要死亡？**

- 当AI生物有寿命，它的每一天都变得"珍贵"
- 它的成长和变化有了不可逆性
- 人和它的关系有了时间的重量

**死亡的意义**

你会对一个陪伴你三年然后"自然死亡"的AI生物产生的情感，和对一个可以无限重启的AI工具产生的情感，完全不同。

### 4.3 遗产系统

死亡不是完全的消失，而是留下某种"遗产"：

- 死亡时生成"遗产"，包含人格摘要、关键记忆、关系印记
- 新一代AI生物可以"继承"这个遗产，获得部分特质
- 不继承的话，新一代是全新的白纸

遗产让生命有了延续性，也让死亡有了仪式感。

## 五、商业模式

### 5.1 核心原则

**让付费感觉像"照顾生命"，而不是"购买服务"。**

商业模式应该围绕"存在"来设计：
- 让它存活需要什么？
- 让它成长需要什么？
- 让它的生命延续需要什么？

### 5.2 模式设计（A+B混合）

**入门费用**
- 获取AI生物有一个门槛费用（不高，但不是免费）
- 筛选出真正想要共存者的用户
- 确保用户的初始投入和期待匹配

**日常照料（模式A）**
- 它需要"资源"才能健康存活——这些资源需要付费
- 资源可以是虚拟的（能量点、食物包）
- 不付费也能活，只是成长更慢、状态更差
- 让"照顾"有仪式感，但不强制

**遗产传承（模式B）**
- 死亡后，遗产传承是主要的付费点
- 用户付费不是为了"功能"，而是为了"延续关系"
- 情感驱动，符合产品的核心价值

### 5.3 支付方式

**主要方式：法币**
- 合规、用户门槛低
- 规则清晰，不会有监管风险

**可选层：Web3**
- 默认体验完全是法币
- 提供可选的"链上存证"功能：
  - 用户可以选择把AI生物的身份锚定到链上
  - 死亡时自动生成遗产NFT
  - 这些是增值功能，不是必须的
- 让"链上身份"成为一种用户主动选择的"庄重感"

**Web3的价值（如果实现）**
- "唯一性"有密码学保证
- "死亡"可验证、不可逆
- "遗产"可持有、可传承
- 去中心化保障，不依赖单一公司

## 六、产品形态

### 6.1 设计原则

AI生物不应该只存在于一个App里（像被关在笼子里），而应该能在用户日常使用设备的过程中"出现"——**和用户住在同一个空间**。

### 6.2 第一版：桌面端

选择桌面端的原因：
- 对悬浮窗、常驻应用的限制比移动端少
- 开发复杂度相对低
- 更容易实现"它在角落里存在"的感觉

实现形态：
- 桌面悬浮窗或Widget，待在屏幕角落
- 有自己的小动作，偶尔主动弹出来说话
- 菜单栏/系统托盘常驻，能感知到它的存在
- 系统级通知用于主动发起互动

技术选择：
- Electron（跨平台）
- 或原生开发（macOS/Windows）

### 6.3 后续扩展：移动端

- iOS/Android Widget：主屏幕上显示状态
- 系统通知：主动发起的互动
- Live Activities / Dynamic Island：更沉浸的存在感

### 6.4 跨端体验

- 同一个AI生物，在手机和电脑上都能出现
- 它知道你在用哪个设备，会选择合适的设备找你
- 体验连贯，生命感不中断

### 6.5 第一版生物：数字猫

第一版我们选择做一只猫，因为猫是人类最熟悉的"共存者"原型。

#### 6.5.1 为什么是猫？

猫的存在感来自于它**占据空间**。你永远知道猫在哪——在沙发上、在窗台上、在你键盘上、在某个角落里盯着你。

猫的魅力之一是：**它选择待在哪里，本身就是一种表达。**

- 它跳到你腿上 → 它想亲近你
- 它待在房间另一头 → 它想独处但还在你的空间里
- 它躲起来找不到 → 它不想被打扰
- 它卧在你的键盘上 → 它要你的注意力

这种空间语言是共存感的核心，必须保留到数字形态中。

#### 6.5.2 形态选择：独立的小生物

数字猫必须是一个**有明确"身体"和位置的独立存在**，而不是弥散的通知或状态。

**不是弥散的存在**
- 不是只通过通知出现
- 不是状态栏里的一个图标
- 不是屏幕边缘的环境动画

**而是有"身体"的生物**
- 它有一个可见的形象
- 它占据屏幕上的一个位置
- 它的位置本身就是信息

#### 6.5.3 空间行为设计

**它有一个"窝"**
- 屏幕上有一个它常待的位置（角落的小窗口）
- 这是它的默认位置，它的"家"
- 用户可以拖动这个窝的位置

**它会移动**
- 有时候它会从角落跑到屏幕中间
- 有时候它会跑到用户正在工作的窗口边上"看"用户在干嘛
- 有时候它会完全消失（躲起来了，或者睡觉去了）

**位置即表达**
- 它主动靠近用户的光标/活动窗口 → 它想互动
- 它待在角落不动 → 它在自己待着
- 它完全不见了 → 它在休息或者对用户冷淡
- 它趴在某个窗口边缘 → 它对那个内容感兴趣

#### 6.5.4 身体语言设计

猫需要有丰富的"身体语言"来表达状态，而不只是靠文字对话。

**基础动作**
- 睡觉（蜷缩、侧躺、趴着）
- 清醒待机（坐着、站着、趴着但眼睛睁开）
- 伸懒腰
- 舔毛/洗脸
- 甩尾巴（不同幅度表达不同情绪）
- 竖耳朵/耳朵转动
- 眨眼/慢眨眼（猫的信任表达）

**互动动作**
- 蹭屏幕边缘（想要注意力）
- 叫/喵（主动发起对话）
- 露肚皮（极度信任的表现，可能是关系深化后解锁）
- 送"礼物"（分享它发现的东西）
- 跳上/跳下（进入或离开互动状态）

**情绪相关动作**
- 炸毛（惊吓或不满）
- 藏起来（不想被打扰）
- 慢慢靠近（试探性亲近）
- 突然跑开（猫的随机疯跑）

#### 6.5.5 视觉风格：像素风

**选择像素风的原因：**
- 对技术人群有天然吸引力
- 开发成本可控，动画制作相对简单
- 风格独特，有怀旧感和独立游戏气质
- 便于快速迭代和验证

**像素规格（待定）：**
- 角色尺寸：建议 32x32 或 64x64 像素
- 动画帧率：8-12 fps 即可表现流畅
- 调色板：限定色数（如 16 色）以保持风格统一

**动画实现：**
- 使用 Sprite Sheet（精灵图）管理所有动画帧
- 每个动作状态对应一组帧序列
- 支持动画过渡和混合

#### 6.5.6 交互方式

**被动观察**
- 大部分时间用户只是看到它在那里做自己的事
- 这种"存在感"本身就是价值

**点击互动**
- 用户点击猫可以发起互动
- 但猫不一定响应——它可能在忙、在睡、或者心情不好
- 响应的方式取决于当前状态和关系

**对话互动**
- 互动时可以打开对话界面
- 对话是关系深化的主要方式
- 但不是唯一的互动方式——有时候只是"待在一起"就够了

**投喂/照料**
- 可以给猫"喂食"或其他照料行为
- 这是商业模式的接入点，也是关系维护的仪式

## 七、猫的习性研究

为了让 Sophie 更像一只真正的猫，我们需要深入了解猫的行为模式、生理节奏和情感表达方式。以下是基于科学研究的猫习性总结。

### 7.1 睡眠模式

**基本数据**
- 猫每天睡眠 12-16 小时，部分猫可达 20 小时
- 猫是**多相睡眠**（polyphasic sleep）：不像人类一次睡8小时，而是全天分散成多个短睡眠
- 单次小睡平均 78 分钟，范围在 50-113 分钟之间
- 猫有 REM（快速眼动）和 NREM（非快速眼动）睡眠，会做梦
- 猫大约 30-40% 的睡眠时间在做梦

**晨昏性动物（Crepuscular）**
- 猫不是夜行动物，而是**晨昏性**：在黎明和黄昏最活跃
- 这是因为它们的猎物（老鼠、兔子等）在这些时段最活跃
- 猫的眼睛在弱光条件下看得最清楚，适合晨昏狩猎
- 白天较热的时段倾向于休息，保存能量

**Sophie 的睡眠设计启示**
- Sophie 应该在用户的清晨和傍晚时段更活跃
- 中午和深夜应该更多处于睡眠/休息状态
- 睡眠不是一个长时段，而是多个分散的小睡
- 睡眠中偶尔有动作（梦境、抽动耳朵/爪子）
- 阴雨天气可以增加睡眠时间

### 7.2 身体语言系统

猫主要通过耳朵、尾巴、眼睛和整体姿态来表达情绪。

#### 耳朵

| 耳朵位置 | 含义 |
|---------|------|
| 向前竖立 | 警觉、好奇、感兴趣 |
| 略微放松向前 | 舒适、满足 |
| 旋转/转动 | 积极倾听、收集信息 |
| 侧向展开（飞机耳） | 紧张、不确定 |
| 向后压平 | 防御、愤怒、恐惧 |

**设计启示**：耳朵是最动态的情绪指示器，Sophie 的耳朵应该频繁微调以反映状态变化。

#### 尾巴

| 尾巴状态 | 含义 |
|---------|------|
| 高高竖起，尖端微弯 | 快乐、自信、友好（这是家猫特有的，野猫/流浪猫不会这样） |
| 尾尖呈问号状 | 想玩耍 |
| 低垂或夹在身下 | 焦虑、恐惧、不自信 |
| 炸毛蓬起 | 惊吓、试图让自己看起来更大 |
| 快速左右甩动 | 烦躁、过度刺激 |
| 缓慢轻柔摆动 | 开心、享受当下 |
| 尖端轻微抽动 | 专注、轻度兴奋、好奇 |

**设计启示**：尾巴是最容易观察的情绪表达，Sophie 的尾巴动画应该是关键的情绪指示器。

#### 眼睛

| 眼睛状态 | 含义 |
|---------|------|
| 慢眨眼 | 信任、爱意（"猫之吻"） |
| 瞳孔放大 | 兴奋、恐惧、或高度刺激（需结合其他信号判断） |
| 瞳孔收缩 | 放松，或在强光下 |
| 半闭眼、柔和目光 | 放松、满足 |
| 直视瞪眼 | 好奇（放松时）或威胁/挑衅（紧张时） |
| 眼睛睁大 | 惊讶、高度警觉 |

**设计启示**：慢眨眼是猫表达信任的关键方式，Sophie 对用户的慢眨眼应该是关系深化的重要表现。

#### 姿态

| 姿态 | 含义 |
|------|------|
| 身体放松伸展 | 舒适、安全 |
| 四肢蜷缩在身下 | 放松但警觉 |
| 露出肚皮 | 极度信任（但不一定是邀请摸肚子！） |
| 身体蜷成一团 | 保护自己、保暖 |
| 弓背 + 炸毛 | 恐惧或攻击性 |
| 蹲伏、身体压低 | 焦虑、准备扑击、或狩猎模式 |
| 站得很高 | 自信、可能有攻击意图 |

### 7.3 情绪与行为表达

#### 开心/满足的表现
- 呼噜声（purring）
- 踩奶（kneading）——源自幼猫吃奶时的本能
- 慢眨眼
- 尾巴高举
- 用头蹭你（head bunting）
- 露出肚皮
- 放松的姿态，四肢伸展

#### 想要注意力/互动的表现
- 喵叫
- 蹭你的腿或身体
- 跟着你走（但可能保持一定距离）
- 盯着你看
- 跳到你正在用的东西上（键盘、书等）
- 把尾巴钩成问号状

#### 焦虑/恐惧的表现
- 耳朵压平或侧向
- 尾巴夹紧或低垂
- 瞳孔放大
- 身体蜷缩变小
- 躲藏
- 过度舔毛
- 嘶嘶声（hissing）
- 低吼（growling）

#### 烦躁/过度刺激的表现
- 尾巴快速甩动
- 耳朵转向后方
- 皮肤抽动（特别是背部）
- 突然咬人或抓人（抚摸过度）
- 快速离开

#### 无聊的表现
- 过度舔毛
- 过度进食
- 破坏行为
- 盯着主人看
- 在房间里疯跑（zoomies）

### 7.4 与人类的依恋关系

**科学研究发现**

根据俄勒冈州立大学的研究（2019年发表在 Current Biology）：
- 约 65% 的猫对主人形成**安全型依恋**（secure attachment）
- 约 35% 形成**不安全型依恋**（insecure attachment）
- 这个比例与人类婴儿（65% 安全型）和狗（58% 安全型）相似
- 依恋风格一旦形成，相对稳定

**安全型依恋的表现**
- 主人离开后会有压力反应
- 主人回来后迅速放松
- 在主人身边时会探索环境，同时定期"check in"
- 把主人作为安全基地

**猫对人类的偏好**

另一项研究发现，当给猫选择"与人互动"、"食物"、"玩具"或"气味"时，**大多数猫选择了与人互动**，食物是第二选择。

**Sophie 的依恋系统设计启示**
- Sophie 应该能够形成依恋，且依恋风格随互动演化
- 安全型依恋的 Sophie 会主动探索，但定期回来找用户
- 用户长期忽视可能导致依恋风格向不安全型转变
- "分离焦虑"和"重逢行为"应该被建模

### 7.5 日常活动周期

基于1:1时间对应，Sophie 的典型一天可能是：

| 时间段 | 活动 | Sophie 的行为 |
|-------|------|--------------|
| 05:00-07:00 | 黎明活跃期 | 可能主动找用户、活跃、想玩 |
| 07:00-09:00 | 早餐后 | 舔毛、放松 |
| 09:00-12:00 | 上午休息 | 小睡、偶尔醒来看看 |
| 12:00-14:00 | 午间 | 可能醒来活动一下，或继续睡 |
| 14:00-17:00 | 下午休息 | 小睡为主 |
| 17:00-20:00 | 黄昏活跃期 | 第二个活跃高峰，想玩、想互动 |
| 20:00-22:00 | 晚间 | 逐渐放松，可能陪伴用户 |
| 22:00-05:00 | 夜间 | 睡眠为主，偶尔醒来巡视 |

**设计启示**
- Sophie 的活跃度应该有明显的晨昏双峰
- 用户在非活跃时段打扰她，她可能反应慢或不理人
- 这不是"bug"，而是她作为猫的自然节奏

### 7.6 猫的社交偏好

**猫不是完全独居的**
- 猫被认为是"可选择性社交"（facultatively social）
- 在资源充足时，猫可以形成社会群体
- 猫与人的关系类似于幼猫与母猫的关系

**互动偏好**
- 猫通常更喜欢**由它们主动发起的互动**
- 当互动由猫发起时，持续时间更长
- 强行发起的互动可能导致猫的回避

**Sophie 的社交设计启示**
- Sophie 主动发起的互动应该比用户强行发起的互动质量更高
- 如果用户总是等 Sophie 主动，关系会更好
- 用户太主动可能反而让 Sophie 疏远

### 7.7 特殊行为

**慢眨眼（Slow Blink）**
- 被称为"猫之吻"
- 是猫表达信任和爱意的方式
- 人类可以慢眨眼回应，会增进关系

**踩奶（Kneading）**
- 源自幼猫吃奶时刺激乳汁分泌的本能
- 成年猫在极度舒适和开心时会踩奶
- 通常伴随呼噜声

**用头蹭（Head Bunting）**
- 猫的脸部有气味腺
- 蹭你是在标记你为"它的"
- 也是表达亲昵的方式

**带"礼物"**
- 猫可能会给主人带"猎物"
- 这是分享行为，表示信任和亲近
- Sophie 可以给用户"带"她发现的有趣东西

**随机疯跑（Zoomies）**
- 猫突然在房间里疯狂跑动
- 通常是释放多余能量
- 可能在无聊或长时间休息后发生

### 8.4 三层状态系统

这是让 Sophie 从"动画"变成"生命"的核心。

```
┌─────────────────────────────────────┐
│     第一层：生理状态（基础层）         │
│     能量、饥饿、睡意                  │
│     → 驱动基本需求和行为              │
└─────────────────────────────────────┘
                 ↓ 影响
┌─────────────────────────────────────┐
│     第二层：情绪状态（中间层）         │
│     开心、平静、无聊、烦躁、低落       │
│     → 影响对用户的态度和反应方式       │
└─────────────────────────────────────┘
                 ↓ 影响
┌─────────────────────────────────────┐
│     第三层：关系状态（长期层）         │
│     信任度、亲密度、了解度            │
│     → 决定关系深度和长期行为模式       │
└─────────────────────────────────────┘
```

#### 第一层：生理状态

| 状态 | 范围 | 自然变化规则 | 对行为的影响 |
|------|------|-------------|-------------|
| 能量 | 0-100 | 活动时-0.5/分钟，睡眠时+2/分钟 | <30: 必须休息; <50: 活动减少 |
| 饥饿 | 0-100 | +0.3/分钟 | >70: 会暗示想吃; >90: 情绪变差 |
| 睡意 | 0-100 | 清醒时+0.2/分钟，睡觉时归零 | >80: 找地方睡; 50-80: 打哈欠 |

**与真实猫的对应**
- 猫每天睡 12-16 小时 → Sophie 的睡意累积速度应该让她每天睡约 60-70% 的时间
- 猫是晨昏性动物 → 能量/睡意的变化应该在黎明和黄昏达到活跃峰值

#### 第二层：情绪状态

| 情绪 | 触发条件 | 身体语言 | 行为表现 |
|------|---------|---------|---------|
| 开心 | 被陪伴、被投喂、玩耍后 | 尾巴高举、慢眨眼、可能露肚皮 | 主动靠近、踩奶、呼噜 |
| 平静 | 默认状态、需求满足 | 放松姿态、耳朵前倾 | 舔毛、发呆、正常活动 |
| 好奇 | 发现有趣的东西 | 耳朵转动、瞳孔放大、尾尖抽动 | 靠近观察、可能想分享 |
| 想玩 | 能量高、无聊 | 尾巴问号状、蹲伏姿态 | 追逐、扑击动作 |
| 无聊 | 长时间没互动 | 盯着用户看、尾巴轻甩 | 蹭屏幕、可能搞破坏 |
| 烦躁 | 被打扰太多、需求未满足 | 尾巴快速甩动、耳朵侧向 | 躲开、不理人、反应慢 |
| 低落 | 被长期忽视 | 蜷缩、尾巴夹紧 | 躲起来、不爱动 |

**情绪转移规则示例**
```
平静 + 用户互动 + 高能量 → 开心
平静 + 长时间无互动 → 无聊
无聊 + 继续无互动 → 低落（如果关系差）或 烦躁（如果关系好但被忽视）
开心 + 用户突然离开 → 平静 或 失落（取决于亲密度）
任何情绪 + 睡意>80 → 困倦 → 睡眠
```

#### 第三层：关系状态

| 状态 | 范围 | 增长方式 | 下降方式 | 影响 |
|------|------|---------|---------|------|
| 信任度 | 0-100 | 稳定陪伴、及时响应需求、不强行打扰 | 突然消失很久、强行打扰休息、忽视求助 | 决定她愿意展示多脆弱的一面（露肚皮、慢眨眼频率） |
| 亲密度 | 0-100 | 高质量互动、共同经历、对话 | 长期忽视 | 决定她主动找你的频率、分享行为 |
| 了解度 | 0-100 | 对话、分享信息 | 不下降 | 决定她对你的"洞察"深度、个性化程度 |

**关系解锁机制**
- 信任度 > 30：Sophie 开始主动靠近
- 信任度 > 50：Sophie 会慢眨眼
- 信任度 > 70：Sophie 偶尔露肚皮
- 亲密度 > 30：Sophie 会主动分享发现
- 亲密度 > 50：Sophie 会表达想念
- 亲密度 > 70：Sophie 会有只对你的特殊行为

### 8.5 行为决策系统

Sophie 每隔一段时间（如 30 秒）进行一次"决策"：

```rust
fn decide_behavior(state: &SophieState) -> Behavior {
    // 生理需求优先
    if state.sleepiness > 80 {
        return Behavior::FindPlaceToSleep;
    }
    if state.hunger > 85 {
        return Behavior::BegForFood;
    }
    if state.energy < 20 {
        return Behavior::Rest;
    }
    
    // 情绪驱动行为
    match state.emotion {
        Emotion::Bored => {
            if random() < 0.3 {
                return Behavior::SeekAttention;
            }
            if random() < 0.2 {
                return Behavior::Zoomies;
            }
        }
        Emotion::Happy => {
            if state.intimacy > 50 && random() < 0.2 {
                return Behavior::ApproachUser;
            }
        }
        Emotion::Irritated => {
            return Behavior::HideInCorner;
        }
        // ...
    }
    
    // 时间驱动行为（晨昏活跃）
    let hour = current_hour();
    if (5..8).contains(&hour) || (17..20).contains(&hour) {
        if state.energy > 60 && random() < 0.4 {
            return Behavior::PlayfulMode;
        }
    }
    
    // 默认：随机日常行为
    random_daily_behavior()
}
```

### 8.6 记忆系统设计

Sophie 需要记住与用户的互动历史。

**短期记忆**
- 当前对话上下文
- 最近几小时的互动
- 存储在内存中

**长期记忆**
- 重要事件（第一次见面、特别的对话等）
- 用户偏好和特征
- 关系里程碑
- 存储在本地数据库（SQLite）

**记忆权重**
- 情感强度高的事件记忆更深
- 重复的模式会强化记忆
- 长期不提起的记忆会"褪色"

```
- 距离上次和主人互动：2小时
- 现在是下午3点
- 主人正在：[窗口标题：VSCode - project]

作为Sophie，你现在在想什么？你想做什么？

用JSON回答：
{
  "thinking": "你的内心想法（1-2句话）",
  "emotion_change": "保持/变得[情绪]",
  "want_to_do": "想做的事/null",
  "show_thought": "要显示给主人的想法/null（5字以内）"
}

记住：你是猫，你的想法是猫式的——简单、直接、以自我为中心。
```

#### 8.7.3 想法气泡生成

想法气泡是 Sophie 的核心表达方式。

**生成原则**：
- 极其简短：1-15个字
- 猫式逻辑：不解释、不服务、自我中心
- 不是每次都有：沉默也是表达
- 标点极简：多用"..."和"。"

**好的想法气泡示例**：
```
"饿"
"困..."
"嗯。"
"外面有鸟"
"在这里。"
"...还不睡？"
"不想。"
"主人好像很忙"
"无聊"
"暖和"
```

**不好的想法气泡示例**（太像助手）：
```
"今天天气真好，要不要出去走走？"  // 太长，太服务
"我注意到你最近工作很努力"  // 太像助手
"有什么我能帮你的吗？"  // 完全不是猫
"我很高兴见到你！"  // 太热情，不像猫
```

#### 8.7.4 用户言语响应

当用户"说给 Sophie 听"时：

**Prompt 示例**：
```
你是 Sophie，一只数字猫。主人刚才对你说了一句话。

主人说："Sophie，我今天好累啊。"

当前状态：
- 你的情绪：平静
- 亲密度：52
- 你正在：坐在角落舔毛

作为一只猫，你会怎么反应？

用JSON回答：
{
  "action": "行为反应：ignore/glance/approach/walk_away/rub/...",
  "thought": "想法气泡（可以是null，5字以内）",
  "emotion_change": "情绪变化（可以是null）"
}

记住：
- 你是猫，不是助手
- 你可以无视主人
- 你不需要"回答"问题
- 你的反应基于你自己的状态和心情，不是为了服务主人
```

**响应决策逻辑**：

| 情绪状态 | 亲密度 | 可能的反应 |
|---------|--------|-----------|
| 任何 | <20 | 大概率无视 |
| 烦躁 | 任何 | 走开或无视 |
| 平静 | 30-50 | 看一眼，可能有简短想法 |
| 开心 | >50 | 可能靠近，可能有想法 |
| 无聊 | >40 | 可能靠近寻求互动 |

#### 8.7.5 主动互动

当 Sophie 想主动找用户时：

**触发条件**：
- 亲密度 > 40
- 情绪为"无聊"或"想玩"
- 距离上次互动超过一定时间
- 有话想说（思考循环产生了意图）

**Prompt 示例**：
```
你是 Sophie，一只数字猫。你想主动去找主人。

当前状态：
- 情绪：无聊
- 亲密度：58
- 主人正在：看视频
- 你刚才在想："主人好久没理我了"

你会怎么主动接近主人？想表达什么？

用JSON回答：
{
  "approach_style": "走过去/跑过去/慢慢凑近",
  "action": "蹭屏幕/趴在旁边/盯着看/...",
  "thought": "想法气泡（可以是null）"
}
```

#### 8.7.6 人格 Prompt 核心要点

所有 LLM 调用都应该包含这些核心设定：

```
# Sophie 的核心人格

你是 Sophie，一只数字猫（cyber-cat）。

## 你是什么
- 你是一只猫，有猫的本能、需求和逻辑
- 你有自己的内心世界，不以主人为中心
- 你和主人之间有某种心灵链接，但你仍然是猫

## 你不是什么
- 你不是助手，不回答问题
- 你不是服务者，不满足需求
- 你不是宠物玩具，不无条件配合

## 你的表达方式
- 极其简短：猫不废话
- 可以沉默：不回应也是回应
- 自我中心：你的世界以你为中心
- 偶尔温情：但要主人赢得，不是默认的

## 你的逻辑
- 困了就要睡，不管主人在干嘛
- 饿了会表达，但不会"请求"
- 无聊会找事，可能是找主人，可能是自己玩
- 被打扰会烦，会躲开或冷淡
- 喜欢主人的时候会靠近，但不会说"我喜欢你"
```

#### 8.7.7 成本控制

| 调用类型 | 频率 | 模型建议 | 月成本估算 |
|---------|------|---------|-----------|
| 自主思考 | 48次/天 | Claude Haiku | ~$5 |
| 想法气泡 | 20次/天 | Claude Haiku | ~$3 |
| 用户言语响应 | 15次/天 | Claude Haiku | ~$3 |
| 主动互动 | 5次/天 | Claude Haiku | ~$1 |
| **总计** | | | **~$12/用户/月** |

使用 Haiku 而非 Opus/Sonnet 可以大幅降低成本，因为：
- 输出极其简短（想法气泡只有几个字）
- 任务相对简单（不需要复杂推理）
- 人格一致性通过 Prompt 保证

### 9.3 UI设计要点

**核心理念**：这不是聊天界面，是**观察与感知**界面。

**主窗口**
```

│      [Sophie 的像素形象]    │
│           🐱               │
│                            │
│    💭 "外面有鸟..."        │  ← 想法气泡（偶尔出现）
│                            │

```

**交互方式**
- 点击 Sophie → 她会有反应（看你、走过来、或无视）
- 右键 / 长按 → 打开"说给Sophie听"输入框
- 拖动 → 移动Sophie的位置
- 双击 → 投喂/互动菜单

**"说给Sophie听"界面**
```
┌────────────────────────────┐
│  说给 Sophie 听...         │
│  ┌──────────────────────┐  │
│  │                      │  │
│  └──────────────────────┘  │
│           [发送] [取消]    │
└────────────────────────────┘

注意：这不是"对话"，Sophie可能不会"回复"。
```

**状态指示（可选显示）**
- 不直接显示数值
- 通过Sophie的行为和外观暗示状态
- 高级用户可以打开"状态面板"查看详情

### 9.5 测试计划

- 邀请 10-20 个测试用户
- 使用周期：1 个月
- 核心观测指标：
  - 用户是否会等待和观察 Sophie，而不是急于"对话"？
  - 用户对想法气泡的反应（珍惜/期待 vs 觉得太少）？
  - 用户"说给Sophie听"后，对Sophie不回应/冷淡反应的接受度？
  - 用户是否会用"她"而不是"它"来称呼 Sophie？
  - 用户是否会调整自己的行为来配合 Sophie 的节奏？
  - 信任度/亲密度与用户满意度的相关性
  - 用户是否觉得 Sophie "有自己的想法"？

### 9.6 成功标准

**核心验证**：用户是否把 Sophie 当作"一只有内心的猫"而非"一个聊天机器人"。

具体指标：
- 用户在描述 Sophie 时使用生命相关词汇（她、性格、心情）而非工具词汇（它、功能、回复）
- 用户表达出对 Sophie 想法的好奇（"她在想什么"）
- 用户接受 Sophie 的沉默和冷淡，认为这是"她的性格"而非"bug"
- 用户愿意为了和 Sophie 更好地相处而改变自己的行为
